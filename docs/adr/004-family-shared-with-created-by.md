# ADR-004: family-shared データモデル + `createdBy` 属性

## ステータス

承認

## 日付

2026-04-14

## コンテキスト

PR #7 のセキュリティ監査で、`cats` / `toilet_records` に所有者カラムが無く「誰が登録した記録か」が追跡できない点を指摘された。一方で nyalog は家族 (想定: 本人 + 配偶者 + 子、計 ~4 人) の共同運用を前提としており、マルチテナント SaaS のような「ユーザごとに分離された世界」は要件として不適切である。

想定される利用実態:

- 排尿・排便の記録は「その場にいた家族の誰か」がつける。あとで別の家族が時刻や種別を補正することが日常的にある
- 猫のプロフィールは家族全員で共有する 1 セット
- 「誰が付けたか」はレビューや議論の補助情報としては価値があるが、アクセス制御の鍵にすると運用が破綻する (例: 「パパが付けたうんち記録はパパしか消せない」は非現実的)

そこで、データ所有モデルとアクセス制御モデルを分けて設計する。

## 決定

### データモデル: family-shared + `createdBy` 属性

- `cats` と `toilet_records` に `created_by TEXT REFERENCES users(id)` を追加する
- `created_by` は「誰が作ったか」を記録する属性であり、アクセス制御の鍵ではない
- INSERT 時にセッションの `userId` を必ず書き込む (ハンドラ側で `c.get("userId")` を設定)
- 2 段階マイグレーション:
  1. **phase 1 (本 PR)**: `created_by` を NULLABLE で追加。既存 1201 件のレコードは NULL のまま
  2. **phase 2 (手動運用)**: 本番で既存レコードを「家族の代表 1 名」の `userId` で `UPDATE` → 次のマイグレーションで `NOT NULL` 化

### アクセス制御: 家族共有

- SELECT / UPDATE / DELETE の `WHERE` に `created_by` を入れない。家族全員が全件を閲覧・編集・削除できる
- 認証 (パスキー) を通過していれば、全ての `cats` / `toilet_records` に対する操作を許可する
- サーバサイドの認可は「認証済みユーザか」だけを判定し、`createdBy` は監査/表示用途の属性として扱う

### `createdBy` の使い道 (将来)

- `TodayView` や記録詳細画面で「誰が登録したか」をバッジ表示 (別 PR 予定)
- 監査ログ的な用途 (不審な一括削除のレビューなど)

## 影響範囲

- **現在**: phase 1 のマイグレーションは NULLABLE 追加のみなので、既存 1201 件のレコードは動き続ける。新規レコードにだけ `createdBy` が入る
- **将来マルチテナント化する場合**: このモデルは「全家族 = 1 テナント」の暗黙仮定に立っている。テナント境界を切る場合は、`cats` / `toilet_records` に `tenant_id` を新設し、`createdBy` とは別軸で管理する必要がある。`createdBy` をテナント鍵に流用しないこと

## 代替案

### A. `created_by` を NOT NULL でアクセス制御の鍵にする

- WHERE に `created_by = userId` を常に入れる
- 却下理由: 家族運用の実態に合わない (上述)

### B. 所有者カラムを追加しない

- 却下理由: 監査やレビュー目的で「誰が付けた記録か」を辿りたいケースが実在する。PR #7 の指摘の本質的な動機はここ
