# ADR-001: プロジェクト開始と技術選定

## ステータス

承認

## 日付

2026-03-27

## コンテキスト

猫の健康管理を一元的に行うWebアプリケーション「nyalog」を開発する。以下の機能を提供する:

- **トイレ記録**: 排泄の回数・状態を記録
- **薬・動物病院の予定管理**: 投薬スケジュールや通院予定を管理
- **ご飯の商品とカロリー表示**: フードの銘柄・カロリー情報を管理

個人的なアプリケーションであり、自分と招待したGoogleアカウントのみがログイン可能とする。

## 決定

### 技術スタック

| レイヤー | 技術 |
|---|---|
| インフラ・ホスティング | Cloudflare Workers / Pages |
| データベース | Cloudflare D1 |
| ORM | Drizzle ORM |
| バックエンドフレームワーク | Hono (RPC) |
| バリデーション | Zod |
| エラーハンドリング | neverthrow |
| フロントエンド | React |
| ビルドツール | Vite+ (`vp`) |
| パッケージマネージャ | pnpm |

### 認証

Firebase Authentication または Google OAuth を採用する。比較調査の上で決定する（別ADRにて記録）。

### コーディング思想

**Domain Modelling Made Functional** の思想に基づき実装する:

- コードはドメインモデルやユースケースが **型・代数的データ型・制約・テスト** で「意味」として表現されていることを重視する
- ドメインロジックは純粋関数として表現し、副作用は境界に押し出す
- neverthrow による Result 型でエラーを型安全に扱う
- Zod スキーマでドメインの制約を表現する

### 開発アプローチ

機能実装の前に基盤を整える:

1. **Cloudflareへのデプロイパイプライン**の構築
2. **CORS設定**
3. **認証基盤**の実装
4. **DBマイグレーション**の仕組み構築

デプロイ環境でのエラーに対応できる状態を作り、機能を順次開発してリリースし、動作確認とフィードバックを得られるサイクルを確立する。

### リポジトリ

- プライベートリポジトリ（Cloudflare Access の AUD タグ等、インフラ情報の露出を避けるため）
- リポジトリ: https://github.com/okayus/nyalog

## 影響

- Cloudflare のエコシステムに依存するため、D1 の制約（SQLite ベース、行サイズ制限等）を受け入れる
- Domain Modelling Made Functional の思想により、型定義とバリデーションの初期コストは増えるが、長期的な保守性と正確性が向上する
- 基盤優先のアプローチにより、初期の機能リリースは遅れるが、以降の開発・デプロイサイクルが安定する
